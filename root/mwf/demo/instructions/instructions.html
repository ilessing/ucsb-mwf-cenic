“Going Mobile With MWF” Demo

In this tutorial, we’ll be creating the mobile version of a desktop site from scratch using the Mobile Web Framework. I’m sure you’re raring to go, so let’s get started.

Getting Started

Download and unzip the example files.

Inside you’ll find a series of HTML pages that constitute a desktop site for a fake UCSB department. Open index.html and familiarize yourself with the fake website; you’ll be creating the mobile version.

After you’re finished, find the m directory. This is where the files for the mobile version of the website will be stored. Within this directory, find the template.html file. Open it up and we’ll get to work.

Exploring the Template

The template includes boilerplate HTML markup that we can use as the basis for all of our MWF-enabled websites. You should keep this template around so that you can just copy-and-paste when creating a new mobile website.

Before we start editing, let’s explore the template. First, take a look at the "viewport" meta element: 

<meta name="viewport" content="height=device-height,width=device-width; initial-scale=1.0; maximum-scale=1.0; user-scalable=no;" />

This tag ensures a consistent experience across various mobile devices. The height=device-height, width=device-width, and initial-scale=1.0 values set the default height, width, and scale of the mobile browser’s viewing area to fit device’s dimensions, preventing the browser from assuming the page was designed for desktop and zooming out.

Similarly, maximum-scale=1.0 and user-scalable=no prevent the User from zooming in and out, which will be unnecessary as our MWF-enabled page will fit snugly in the browser’s viewing area.

Next, let’s check out the "format-detection" meta element: 

<meta name="format-detection" content="telephone=no" />

This little tag prevents modern mobile browsers from attempting to convert numbers to telephone number links that can be used to start a phone call. As you’ll see, we’ll explicitly mark our telephone numbers, so we won’t need the browser’s help.

Alright, let’s dig into the two most important elements: 

<link rel="stylesheet" href="http://mwf.library.ucsb.edu/assets/css.php" type="text/css" />
<script src="http://mwf.library.ucsb.edu/assets/js.php" type="text/javascript"></script>

With only these two elements, we’ve now enabled the Mobile Web Framework. The first element imports the Framework’s CSS while the second element imports the Framework’s JavaScript.

You may have noticed that we are not directly referencing a CSS file (.css) or a JavaScript file (.js); instead, we’re referencing a PHP (.php) file. This is the magic of the Framework: when the browser accesses these PHP handlers, they dynamically return CSS and JavaScript appropriate to the capabilities of the device. In other words, a newer device will recieve the full experience (touch transitions, gradients, etc.), while an older mobile phone will recieve a slightly degraded experience more appropriate for their device.

The important thing to note here is that you, as the developer, are only providing one set of markup and the framework is handling the messy details of detecting a device’s capabilities and serving the appropriate resources.

Developing Locally

Before we start, we’ll need to do one thing first: visit UCSB’s MWF website. Just pop it open in a browser. Why? Well, as you know, the Framework detects the capabilities of a User’s device and serves the appropriate CSS and JavaScript. It accomplishes this by storing information in a cookie. Generally, when a User visits a website powered by UCSB’s instance of MWF, the User will be redirected to UCSB’s MWF website and then redirected back to their destination.

However, when developing locally on our computer, the Framework can’t perform this redirect loop. So, we just need to hop over to UCSB’s MWF website and grab the cookie.

Also, just a quick note: if you get lost within the instructions, use the “View Full Source” links to view the entire markup for a page.

Alright, enough talk, let’s get to work.

Create an “About” Page

The template file is great and all, but it doesn’t contain anything interesting. Let’s create a full-fledged “About” page that we can use as a template for the rest of the sections of our mobile website. First, copy the template.html file and create an about.html file.

Open up about.html in a text editor and add the following between the body tags: 

<h1 id="header">
   <img alt="UCSB" src="http://mwf.library.ucsb.edu/assets/img/ucsb-header.png" />
   <span>About</span>
</h1>

You should also change "Section Title" in the title element to "About".

Now, pop open about.html in a browser and check out your handiwork: you’ve added a beautiful header bar to the first section of your first MWF-powered website. The header CSS ID references styling rules provided by the Framework and the UCSB header image is a standardized graphic stored on UCSB’s MWF servers.

Below the header, add the following content area: 

<div class="content-full content-padded">
   <h1>About Us</h1>
   <div>
      Founded in 2012 with the purpose of creating a fake department website for a mobile website demo.
   </div>
</div>

This adds a basic content block. To signal to the Framework that the div is a content block, we tack on the content-full CSS class, which sets up some basic styling. The content-padded CSS class adds the rounded corners and the padding around the content area. Finally, if you view this in a browser, you’ll notice that the Framework automatically styles the h1 element as a title bar with a gray gradient background and large, bolded text.

Next, we’ll add a button to return to the home page (which we’ll create in just a little bit): 

<a class="button-full button-padded" href="index.html">Back to Home</a>

Similarly to the content block, the button-full CSS class signals to the Framework that this a element is a button and the button-padded class adds rounded borders and padding. Let’s wrap up this page with a footer: 

<div id="footer">
   <p>
      &copy;2012 The Regents of the University of California, All Rights Reserved
      <br />
      <a href="http://mwf.library.ucsb.edu/mwf/termsofuse.php">Terms of Use</a>
      | <a href="http://mwf.library.ucsb.edu/mwf/accessibility.php">Accessiblity</a>
      <br />
      <a href="../index.html?override_redirect=1">View Full Site</a>
   </p>
   <br />
   <p class="mwf-credit">
      Powered by the <a href="http://mwf.ucla.edu">Mobile Web Framework</a>
   </p>
</div>

You know the drill by now: the header CSS ID and mwf-credit class reference some footer-specific styling from the Framework. We’ve also added some links here to standard, mobile-ready Terms of Use and Accessibility pages.

However, there is something new here you haven’t seen before: the “View Full Site” link. You’ll notice the URL for this link contains an ?override_redirect=1 at the end. As you’ll see later on, the Framework will automatically redirect mobile Users from the desktop to the mobile site. However, if we allow mobile Users to override this behavior and return to the desktop site, we don’t want to redirect them back to the mobile site. To avoid this redirect loop, we tack on a override_redirect GET variable to the URL to tell the Framework to temporarily disable the redirect behavior for the User.

You’re finished! You’ve created a MWF-enabled “About” section! Pat yourself back and we’ll jump into creating a home page.

Create a “Home” Page

We’re going to add some special styling to our home page but, for the most part, our “About” page and “Home” page will share a similar template, so go ahead and make a copy of about.html named index.html. Open it up and add the front-page CSS class to the body element: 

<body class="front-page">

This signals the Framework to add some home page-specific styling to the page, including a larger, brighter banner at the top. Next, replace the h1 element (with the header CSS ID) with this div: 

<div id="header">
   <img alt="UCSB" src="http://mwf.library.ucsb.edu/assets/img/ucsb-header-front.png" />
</div>

This replaces our header from the “About” page with a larger banner image for our home page. You might have noticed we just completely removed the title of the page, though, so let’s put it back by adding this right beneath the header div you just added: 

<div class="content-full content-padded">
   <h1>Office of Lorem Ipsum Mobile</h1>
</div>

This is just a regular content block as you’ve seen before, but without any content and only a h1 element, it also functions as a nice, bold header for our home page.

Alright, now it’s time to add our navigation menu. Replace the main content block from your “About” template with this: 

<div class="menu-full menu-front menu-detailed menu-padded">
   <ol>
      <li><a href="about.html">About</a></li>
      <li><a href="photos.html">Photos</a></li>
      <li><a href="visitus.html">Visit Us</a></li>
      <li><a href="contact.html">Contact</a></li>
   </ol>
</div>

Although this example presents new markup and new CSS classes, it’s the same basic principle we’ve followed throughout the tutorial: add a few specific CSS classes to your HTML and the Framework’s dynamic styling handles the rest. menu-full is the basic menu class, menu-front adds some extra padding within the menu items to create a larger menu for the home page, menu-detailed aligns the menu items to the left (rather than centering them), and menu-padded adds, you guessed it, padding and rounded corners.

To clean things up, remove the “Back to Home” button and, of course, don’t forget to change the “About” in the title to “Home”.

Oh, look, you’re already finished with the home page. Now that you’ve created a section and a home page, why don’t you start redirecting Users from your desktop to your mobile site?

Add a Mobile Redirect

Adding a mobile redirect is pretty straightforward, you just need to add a single script include. Let’s add it to the home page of the desktop site. Open index.html in the main directory of our example files (not the m directoy), and add the following to the head: 

<script type="text/javascript" src="http://mwf.library.ucsb.edu/assets/redirect/js.php?m={YOUR_MOBILE_URL}"></script>

Unfortunately, because we’re working locally, we don’t have a live URL to set as our redirect script’s destination. In a live site, you would want to replace {YOUR_MOBILE_URL} with the URL-encoded URL of your website.

But, suspend disbelief with me and pretend that we just put in a working mobile redirect, alright? If we had, this script would be automatically redirecting Users from our mobile website to the specified URL.

Now that you’re (theoretically) redirecting Users, you should probably finish up the rest of your mobile site. Let’s start with the “Visit Us” page.

Create a “Visit Us” Page

Create another copy of about.html and name it visitus.html. Replace the main content block with the following: 

<div class="content-full content-padded">
   <h1>Visit Us</h1>
   <div>
      <img width="600" height="320" alt="Map to the Office of Lorem Ipsum" src="">
   </div>
   <div>
      <strong>Office of Lorem Ipsum</strong>
			
      <br />
			
      4256 Donec Luctus Ave.<br />
      Aliquam, CA 95678
			
      <br />
      <br />
			
      <a href="tel:805-893-3592">(805) 893 3592</a>
   </div>
</div>

Ignore that our image has no src value; we’ll fix that in just a second.

First, notice that, unlike our “About” page, this content block contains two child divs: the first div contains the map, while the second div contains the address information. The Framework recognizes this as two sections within a content block and adds a divider line to separate them.

Second, check out that telephone number link. Remember earlier when we said that we would disable the browser automatically detecting phone numbers and turning them into links because we would handle it ourselves? Well, we’re doing that with the tel prefix in the href attribute. This prefix signals to the browser that this is a link that should initiate a phone call.

Now let’s fix that image. For this example, we are going to use the Google Static Maps API. With the Google Static Maps API, we construct a URL that contains information about a location and then Google’s servers return a static image of the location. Our URL will contain the following parts:

  • Base URL: http://maps.googleapis.com/maps/api/staticmap?sensor=false
  • Zoom Level: &zoom=15
  • Map Size: &size=600x320
  • Target Location: &markers=34.41369,-119.843019

Altogether, here’s our static map URL:

http://maps.googleapis.com/maps/api/staticmap?sensor=false&zoom=15&size=600x320&markers=34.41369,-119.843019

Since we’ll be using the URL within our HTML, we’ll want to encode those ampersands (swap & with &amp;):

http://maps.googleapis.com/maps/api/staticmap?sensor=false&amp;zoom=15&amp;size=600x320&amp;markers=34.41369,-119.843019

Now we can pop that URL into our image src: 

<img width="600" height="320" alt="Map to the Office of Lorem Ipsum" src="http://maps.googleapis.com/maps/api/staticmap?sensor=false&amp;zoom=15&amp;size=600x320&amp;markers=34.41369,-119.843019" />

Boom: you’ve just added a map to your page! Next, let’s put together a "Photos" page.

Create a “Photos” Page

Just like before, copy about.html and name the new file photos.html. Replace the main content block with the following: 

<div class="content-full content-padded">
   <h1>Photo Gallery</h1>
   <div>
      <img src="http://web.chem.ucsb.edu/~bazangroup/UCSB-from-air.jpg" />	
   </div>
</div>

You’ve seen all of this before: a basic content block with a title and an image. Try opening this page in a browser and resizing the window. You’ll notice the image shrinks to fit the width of the browser. This is wonderful, but Users on a mobile device will still download the full-scale image. We need some way to provide an image appropriately resized for the viewing area of the User’s device.

Thankfully, the Framework has already solved this problem with the Image Compressor. Similar to the Google Static Maps API, we construct a URL that contains information about how we want to resize the image and the Framework handles the rest.

Let’s put together the URL for our dynamically resized image. You’ll need to do some hardcore URL encoding for this, so go ahead and open up a URL encoder/decoder tool in another window. Alright, our URL will contain the following:

  • Base URL: http://mwf.library.ucsb.edu/assets/min/img.php?
  • URL-encoded Image Souce: img=http%3A%2F%2Fweb.chem.ucsb.edu%2F~bazangroup%2FUCSB-from-air.jpg
  • Percentage of User’s Browser Width: &browser_width_percent=80
  • Maximum Width: &max_width=400

Putting this altogether (and also URL-encoding the ampersands):

http://mwf.library.ucsb.edu/assets/min/img.php?img=http%3A%2F%2Fweb.chem.ucsb.edu%2F~bazangroup%2FUCSB-from-air.jpg&amp;browser_width_percent=80&amp;max_width=400

Just swap out the original image source with this one and voilá, you’ve added an image that will dynamically resize to fit the needs of the User’s device.

While we’re here, why don’t we add a couple buttons to let the User navigate through a gallery of images? Add this below the main content block: 

<div class="button-full button-padded">
   <a href="#" class="button-first button-light">Previous Image</a>
   <a href="#" class="button-last button-light">Next Image</a>
</div>

You’ve seen buttons before, but there are a couple neat things happening here.

First, you’ll notice that there are two links within this button div, rather than a single link as seen in previous examples. Along with that, the first link has a button-first CSS class, while the second link has a button-last class. With these classes, the Framework will style these buttons by placing them side-by-side, with the first button on the left and the second on the right.

We also added a button-light class to the links, which will provide a “softer” look and feel to our buttons, to help differentiate them from the other buttons on the page.

We’ll want to keep moving forward, so let’s leave he number signs (#) instead of URLs for right now. When you come back to this page after finishing the tutorial, you can change these links to point to other pages with more images.

Create a “Contact Us” Page

Phew, alright, so you’ve created a full-featured mobile website with a basic “About” page, a home page with a mobile-friendly navigation menu, a “Visit Us” page containing a static Google map, and a “Photos” page with a dynamically-resized image. Let’s cap it off with a “Contact Us” page.

You’re a pro at this by now: copy about.html and name the new file contact.html. Replace the main content block with the following: 

<form class="form-full form-padded">
   <label for="contact-name">Name</label>
   <input type="text" id="contact-name" name="name" />
		
   <label for="contact-email">E-mail</label>
   <input type="text" id="contact-email" name="email" />
		
   <label for="contact-message">Message</label>
   <textarea id="contact-message" name="message"></textarea>
		
   <input type="submit" value="Send Message" />
</form>

There’s nothing really surprising going on here: we’ve just created a regular HTML form and tacked on form-full and form-padded CSS classes (I’ll let you guess which one sets up the basic styling for the form and which one adds rounded corners and padding).

Open this up in a browser and check out what you’ve come to expect at this point: the Framework has completely styled the form for you.

Congratulations!

Nice work! You’ve created a full-featured, MWF-powered mobile website!

As you can see, MWF makes mobile web development easy by handling all of the nitty-gritty details for you and providing helpful utilities, like the Image Compressor. For more information, visit the MWF Wiki.
